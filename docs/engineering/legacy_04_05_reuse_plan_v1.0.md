## 3. 当前判断

这几份文档的特点很明确：

### 3.1 它们仍然是“在用文档”

原因：

- 版本比 `90_历史归档` 里的副本更新
- 已经包含模型路由增强、runtime 分层预留、里程碑和任务拆解
- 内容明显不是简单历史副本，而是后续又持续补充过

因此，不应把根目录 `04 / 05 / 06` 直接按归档废文档处理。

---

### 3.2 它们已经不适合作为当前主线总纲

原因：

- 叙事中心仍是“多 Agent 长记忆 IM 接入底座”
- 首版目标里仍混入多 Agent forum、Telegram、多模态、模型路由等较重范围
- 任务拆解的 Epic 粒度面向的是旧功能架构，而不是当前已经收敛后的“单 Agent + 长记忆角色 + IM 接入”
- 其中不少模块是长期预留或未来扩展，不适合继续作为当前近期执行清单

因此：

- 可以继续参考
- 不应继续作为当前 Phase 1 主执行文档

---

## 4. 更合适的角色定位

## 8. 对 06 的具体处理建议

## 8.1 当前判断

`06` 和 `04 / 05` 最大的不同在于：

- 它聚焦的是记忆结构本身，而不是整个平台功能面
- 它已经明显收敛到了 Layer A / B / D 的现实问题
- 它多处判断与当前新主线是同向的，而不是相反的

例如其中这些判断，当前仍然有较高价值：

- `Role Core` 的最小 contract
- `Structured Long-Term Memory` 的 ontology
- `Scope / Validity / Status / Update / Recall` contract
- 知识层与记忆层的边界
- Layer D 处于观察期而非实现期
- 当前不为未来世界层过度设计

因此，`06` 不应简单降级成“旧架构参考库”。

---

## 8.2 更合适的使用方式

我建议把 `06` 视为：

- memory layer 方向的过渡主参考文档
- 新版 `memory_layer_design_v1.0.md` 的主要素材来源
- 当前主线中“最值得继续吸收”的旧文档之一

更具体地说：

- `06` 里的 Layer A / B / D 语言可以继续用
- 但应迁移成更贴当前项目结构的新文档表达
- 不建议继续把 `06` 本身当成最终执行文档

## 12. 当前结论

根目录的 `04 / 05 / 06` 更合适的使用方式，不是：

- 继续当主线执行文档
- 也不是立刻当废弃归档

而是：

**按不同角色视为过渡期的设计资产。**

其中：

- `04` 更偏技术参考库
- `05` 更偏任务参考库
- `06` 更偏可继续吸收的记忆设计资产
- 当前真正要执行的内容，应逐步迁移到新的 `docs/strategy / architecture / engineering / product` 体系中

这样既不会丢掉历史积累，也不会让当前主线继续被旧架构牵着走。
