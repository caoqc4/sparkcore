# 虚拟伴侣产品端功能设计 v2.0

## 0. 文档定位

本文档用于在以下前提下，正式定义 SparkCore 第一阶段虚拟伴侣产品的产品端设计方向：

- SparkCore 当前主线仍是“技能底座 + 记忆层升级”
- 第一阶段产品形态为 **IM 内 bot**
- 网站承担 **角色配置、记忆管理、关系控制台、补充聊天入口** 的职责
- 第二阶段将进入 **移动端事务推进 app**，届时虚拟伴侣模块需要可迁移、可复用

本文档不是通用 AI 聊天产品 PRD，也不是纯助理型产品 PRD。

本文档聚焦的是：

> **一个 IM-native 的虚拟伴侣系统：IM 负责高频关系互动，网站负责角色配置、记忆管理、关系控制与补充对话，底层由统一角色核、长记忆与轻技能层驱动。**

---

## 1. 一句话定义

第一阶段产品不是“有人设的工具 bot”，也不是“功能很多的事务助理”，而是：

> **一个以伴侣为先、助理为辅的 IM 原生长期陪伴产品。**

它的核心价值不是“会做很多事”，而是：

1. 像同一个存在
2. 记得住用户与关系
3. 能在 IM 中自然持续互动
4. 用户可在网站上查看、纠正、管理这段关系的长期状态

---

## 2. 产品边界

## 2.1 当前明确要做的

第一阶段要验证的，不是泛助理价值，而是以下闭环：

```text
创建角色 / 领取角色
→ 绑定 IM
→ 开始持续聊天
→ 伴侣逐渐表现出“记得住我”“记得我们”
→ 用户在网站上查看 / 修正 / 管理记忆与角色
→ 用户持续复访与回到同一角色继续互动
```

## 2.2 当前不作为主目标的

本阶段不以以下目标为主：

- 不以复杂任务自动化为主卖点
- 不以大量技能数量为主卖点
- 不以多角色广场、复杂社区生态为主
- 不以多 Agent 协作为主场景
- 不以重度主动提醒 / 强运营触达为第一优先级
- 不以复杂人格搭建器为核心入口

## 2.3 产品定位收口

建议正式收口为：

- **主定位：虚拟伴侣**
- **副定位：具备轻助理能力的长期陪伴角色**

即：

- 伴侣是产品心智中心
- 助理能力是体验增强层，不反向定义产品

---

## 3. 三端职责划分

虽然第一阶段没有移动端 app，但为了第二阶段可迁移，需要从现在开始把职责分清。

## 3.1 IM 端：关系发生地

IM 是第一阶段的主入口，也是用户最自然的高频互动场景。

IM 端主要负责：

- 日常聊天
- 情绪陪伴
- 轻量问答
- 轻量记忆写入
- 关系连续性承接
- 用户自然表达生活流信息

IM 端不承担：

- 复杂设置
- 深度记忆管理
- 复杂角色编辑
- 重控制台功能

一句话：

> **IM 是生活流入口，不是后台。**

## 3.2 网站端：关系控制台

网站不是单纯的“领取页”或“设置页”，而应被定义为：

> **关系控制台 + 深度补充对话入口**

网站端主要负责：

- 角色配置
- 角色资料查看与调整
- 记忆中心
- 同步查看 IM 聊天记录
- 补充型长对话
- 关系轨迹查看
- 数据与隐私控制

网站不是第一阶段的主关系入口，但它是：

- 用户信任建立的关键位置
- 长记忆可见/可控的主要承载面
- 后续迁移到移动端 app 的结构预演

## 3.3 第二阶段移动端 app：事务主场 + 伴侣模块承接

第二阶段的移动端事务推进 app，不应重新发明一套 companion 系统。

而应复用：

- 角色核
- 记忆体系
- 线程状态
- 主动性策略抽象
- 关系与偏好层

也就是说：

> **第一阶段的 companion system 是未来事务 app 的人格层与关系层底座。**

---

## 4. 核心设计判断

## 4.1 关于 IM 与网站聊天同步：采用“线程状态双端共享 + 消息表面单向同步为默认策略”

你补充的判断是对的：

**第一阶段不建议把 IM 和网站做成完全对等的实时双端镜像聊天。**

更合理的策略是：

### 默认策略
- **IM → 网站：同步可见**
- **网站 → IM：不做消息镜像推送**

### 但底层仍需共享
- 同一角色
- 同一 canonical thread
- 同一长期记忆
- 同一线程状态

也就是说：

#### 用户在 IM 说的话
- 网站聊天页能看到
- 网站控制台能基于这些内容展示记忆与关系轨迹

#### 用户在网站说的话
- 这些内容进入同一条关系线程
- 会影响角色后续状态、记忆、线程连续性
- **但不需要把网站上的消息“反向转发成 IM 消息气泡”发回 IM**

这是一个很重要的区分：

> **状态同步是双向的；消息表面的镜像同步不一定双向。**

这样设计的好处：

1. 避免 IM 被网站消息“刷屏”
2. 避免用户对“为什么网页里说的话要回灌到 IM”产生困惑
3. 保持 IM 是主关系入口的产品心智
4. 网站仍保留“补充长对话”能力
5. 未来迁移移动端时，不需要改变底层线程模型

### 结论

第一阶段建议采用：

- **单一 canonical thread**
- **底层状态双向共享**
- **UI 层消息镜像默认单向：IM → 网站**

---

## 4.2 角色创建：少模板、强角色核

你补充的判断成立。

第一阶段不建议做大量模板。

建议只保留两类：

1. **伴侣**（主模板）
2. **助理**（副模板）

并明确：

- 首页与主路径以“伴侣”为先
- 助理只是另一种低情感密度模式，不与伴侣等权竞争

### 为什么不建议一开始上太多模板

因为当前阶段真正需要验证的是：

- 长期关系是否成立
- 角色连续性能否成立
- 记忆中心是否能增强信任
- IM 入口是否适合承载长期陪伴

而不是：

- 市场上是否有 8 种不同人格模板

### 模板设计原则

模板只提供：

- 初始角色设定
- 初始语气
- 初始关系模式
- 初始边界

模板不承担：

- 全部人格搭建能力
- 复杂世界观设定器
- 大规模角色市场

### 角色核比模板更重要

第一阶段真正要做强的是角色核，而不是模板数量。

角色核至少应包含：

- `name`
- `archetype`
- `tone`
- `relationship_mode`
- `proactivity_level`
- `boundaries`
- `core_principles`

---

## 4.3 记忆中心：网站端第一重心

这一点你已经同意，而且我建议在文档里明确写成：

> **记忆中心是第一阶段网站端最核心的产品模块。**

### 为什么记忆中心重要

因为 IM bot 最容易被用户误解为：

- 一个带人设的聊天机器人
- 一个短上下文的玩具角色
- 一个只能表面亲密的对话工具

真正能把它拉回“长期关系产品”的，是记忆中心。

记忆中心解决三件事：

1. 让用户看到“它确实记住了什么”
2. 让用户有能力纠正“它记错了什么”
3. 让长期关系不只是模型黑箱，而是可管理状态

### 第一阶段建议的记忆中心结构

建议至少分成四类：

1. **关于我**
   - 稳定资料
   - 偏好
   - 习惯
   - 长期约束

2. **关于我们**
   - 重要共同事件
   - 关系阶段
   - 重要承诺
   - 被反复提起的互动片段

3. **待跟进**
   - 用户明确说后续还要聊/还要继续的事情
   - 约定再次提起的事项
   - 当前未闭环的话题

4. **已纠正 / 已删除**（可简化）
   - 用户否认或删除的记忆
   - 便于后续降低误召回

### 第一阶段建议支持的操作

- 查看
- 删除
- 纠正
- 置顶
- 标记“不重要”

不建议第一阶段就做：

- 复杂图谱可视化
- 太多自动标签
- 记忆版本树可视化

---

## 4.4 主动性：第一阶段不做重主动触达，只预留“轻回访能力”

你对“主动性为什么要有”的疑问很合理。

这里要把三件事分开：

### A. 强主动提醒 / 主动运营触达
例如：
- 系统自己决定每天找用户聊天
- 高频 push
- 未经明确许可的情绪运营式唤回

**第一阶段不建议做。**

### B. 事务型主动提醒
例如：
- 明天 9 点提醒用户
- 到点催办待办事项

这更接近第二阶段事务推进 app 的能力。

**第一阶段不作为核心功能。**

### C. 轻量关系型回访
例如：
- 用户明确说“明天继续聊这个”
- 用户说“我周五有面试”
- 角色在合理边界内，于约定时间点轻量提起

这才是第一阶段最值得保留的“主动性雏形”。

### 为什么第一阶段仍建议预留轻回访

因为它不是“提醒功能”，而是：

> **关系连续性的一种表现。**

如果完全没有任何回访能力，那么伴侣的长期性会更像：

- 我来找你时你才存在
- 脱离当前轮次后什么都不延续

这会削弱“长期陪伴”的产品感。

### 但第一阶段的控制原则是

- 不自动骚扰
- 不做高频主动消息
- 不做复杂调度系统
- 只做**用户明确授权或明确约定的轻量回访**

### 因此建议文档里这样定义

第一阶段：
- **P0 不做系统级主动外呼**
- **P1 可做“约定式轻回访”**
- **P2 再考虑与事务推进体系打通的主动性机制**

---

## 4.5 与 SparkCore 记忆层升级方案的关系

你上传的记忆层升级方案 v0.1 / v0.2 已经把很多关键底层前提钉住，这对 companion 产品设计非常重要。

第一阶段 companion 产品不应绕开这套记忆结构，而应直接基于它来收口。

### 当前最相关的底层共识

应直接继承以下判断：

1. **Profile / Long-term Memory / Thread State / Recent Raw Turns 分层**
2. **记忆不是日志仓库，而是长期状态系统**
3. **更新 / 失效 / 时间语义要内建**
4. **Thread State 是一等层，不能混进长期记忆**
5. **Core Memory Kernel 与 Scenario Pack 分离**

### 对 companion 产品最有价值的不是全部能力，而是以下四项

1. `Static / Dynamic Profile`
2. `Relationship / Event / Preference` 的结构化长期记忆
3. `ThreadStateRecord`
4. `Context Assembly Engine`

### Companion 可视为第一阶段最先落地的 Scenario Pack

即：

- SparkCore 记忆层是通用核心
- Companion Product 是第一个正式产品化的 `Scenario Memory Pack`

因此 companion 产品端设计需要主动兼容：

- Relationship Profile
- 事件级记忆
- 轻量回访点
- 不同 channel 的统一线程模型

## 5. 第一阶段产品结构

## 5.1 用户路径（建议版）

```text
首页进入
→ 选择伴侣 / 助理模板
→ 完成基础配置
→ 领取角色
→ 绑定 IM
→ 开始在 IM 聊天
→ 网站可看到同步聊天与记忆中心
→ 用户逐渐感知“这个角色记得我”
→ 用户回到网站修正 / 管理记忆与关系
→ 长期留存
```

## 5.2 产品骨架

第一阶段建议只围绕 5 个主模块展开：

1. 角色模块
2. IM 聊天模块
3. 网站同步聊天模块
4. 记忆中心
5. 隐私与关系控制模块

## 6. P0 / P1 / P2 功能树

## 6.1 P0：必须先做的功能

目标：

> **验证 IM-native companion 是否能建立“同一个角色 + 记得住 + 可管理”的基础关系闭环。**

### P0-1. 角色创建（轻量）

支持：
- 选择模板：伴侣 / 助理
- 设置昵称
- 选择语气/风格（少量选项）
- 设置关系模式（陪伴型 / 更轻助手型）
- 选择主动性等级（默认低）

不支持：
- 复杂自定义人格搭建器
- 世界观编辑器
- 多模板 marketplace

### P0-2. IM 绑定与主入口聊天

支持：
- 绑定单一 IM 通道（或首发 1~2 个）
- 从 IM 发起对话
- 角色持续存在
- 多次回来还是同一个角色

### P0-3. 网站聊天页（补充型）

支持：
- 查看 IM 同步过来的聊天记录
- 在网站继续同一角色的聊天
- 网站聊天内容进入同一 canonical thread
- 不把网站消息反向镜像到 IM

### P0-4. 记忆中心 v1

支持：
- 查看“关于我”
- 查看“关于我们”
- 查看“待跟进”
- 删除记忆
- 纠正记忆
- 置顶记忆

### P0-5. 统一线程模型

支持：
- 同一角色跨 IM / 网站共用一条主关系线程
- channel 作为事件来源标签存在
- 长期记忆统一写入
- thread state 统一更新

### P0-6. 轻技能集

只建议开放少量与关系连续性强相关的技能：

- 保存记忆
- 标记后续再聊
- 简单提醒/约定
- 基础信息查询
- 基础日程/待办记录（若底座已具备且很轻）

不建议 P0 做：
- 大量工具接入
- 复杂 workflow 自动化
- 多步事务执行编排

### P0-7. 隐私与边界控制

支持：
- 清空记忆
- 删除单条记忆
- 开关网站聊天
- 开关回访权限（若有）
- 调整角色模式
- 基础安全兜底

## 6.2 P1：建议尽快规划但不必首发做满

目标：

> **增强“关系连续性”与“控制感”，但不把产品拖向过重。**

### P1-1. 关系轨迹页

展示：
- 最近重要互动节点
- 最近被记住的重要事件
- 最近待跟进事项

### P1-2. 约定式轻回访

仅在以下情况下启用：
- 用户明确说“到时候提醒我 / 再问我”

支持：
- 手动设定一个轻回访点
- 回访消息可在 IM 中出现

### P1-3. 角色资料页增强

支持：
- 角色设定查看
- 核心关系模式调整
- 语气/边界调节
- 角色摘要

### P1-4. 记忆中心增强

支持：
- 更多分类
- 记忆来源说明
- 低置信记忆提示

产品侧开始更显式地承接底层结构：
- relationship_stage

## 8. 数据与状态层的产品约束

本节不是技术实现文档，而是产品必须遵守的约束。

## 8.1 角色对象独立于通道

角色不是 Telegram 角色 / Web 角色 / App 角色。

而是：

> **一个独立存在的 companion entity，通道只是它的接入方式。**

## 8.2 线程状态与长期记忆必须分开

产品层不要把这两者混起来展示或使用。

- 线程状态：当前聊到哪
- 长期记忆：长期记住什么

这会直接影响：
- 网站聊天续聊体验
- 待跟进事项展示
- 后续事务推进 app 的迁移

## 8.3 记忆必须允许更新、纠正、失效

产品层不要把记忆当“越多越好”。

需要承认：
- 有些记忆会变旧
- 有些记忆会被用户纠正
- 有些记忆只在某阶段有效

## 8.4 网站聊天不是第二套系统

网站聊天必须与 IM 的主关系线程共享：
- 角色核
- 长期记忆
- thread state

不能成为另一个“平行人格”。

## 9. 核心设计原则

### 原则 1：伴侣优先，助理其次
产品心智先立住“陪伴关系”，再承接轻助理能力。

### 原则 2：IM 优先，网站承接控制与深化
IM 是高频关系入口；网站是控制台与补充对话主场。

### 原则 3：少模板，强角色核
不靠模板数量竞争，而靠角色一致性和长期连续性。

### 原则 4：记忆中心先于复杂 UI 表现
先让记忆可见、可改、可信，再考虑复杂可视化。

### 原则 5：状态同步比消息镜像更重要
底层线程与记忆统一，比表面双端消息完全镜像更重要。

### 原则 6：第一阶段不做重主动运营
只保留轻量、可解释、用户授权的关系型回访能力。

### 原则 7：技能层克制
技能为关系连续性服务，而不是抢占产品定义。

---

## 10. 当前不建议提前做的内容

1. 大量角色模板
2. 复杂角色广场 / 社区
3. 重度主动消息系统
4. 多 Agent 协作 companion
5. 复杂 3D / 视觉化人格表现
6. 复杂关系图谱可视化
7. 大量 workflow 工具接入
8. 把网站和 IM 做成完全对等实时镜像聊天

---

## 11. 首版核心指标建议

第一阶段不应该只看 DAU / 发送消息数。

更应该看：

### 关系建立类
- 绑定 IM 的转化率
- 领取角色后的首周复访率
- 回到同一角色继续聊的比例

### 记忆感知类
- 用户对“它记得我”的主观评分
- 用户进入记忆中心的比例
- 记忆纠正 / 删除后的留存变化

### 连续性类
- 同一角色跨天续聊率
- 网站聊天与 IM 聊天的协同使用率
- thread continuity 主观评分

### 风险类
- 误记忆率
- 串角色 / 串线程率
- 用户对过度主动的投诉率

---

## 12. 与第二阶段事务推进 app 的兼容约束

为了后续迁移顺滑，第一阶段需要提前遵守以下约束：

1. Companion 角色对象独立存在
2. 记忆 schema 不依赖 IM 通道结构
3. thread state 为独立层
4. 轻回访机制抽象为策略层，不绑死在 IM
5. 技能调用走统一 action 层
6. 网站控制台未来可以自然映射成 app 的 companion 模块 / 个人状态页 / 记忆中心

---

## 13. 最终结论

第一阶段虚拟伴侣产品的正确做法不是：

- 做一个功能很多的 bot
- 做一个模板很多的角色市场
- 做一个会聊天的轻工具

而是：

> **做一个 IM-native 的长期陪伴角色系统：IM 是关系发生地，网站是关系控制台，角色核 + 长记忆 + 轻技能层是底座，记忆中心是网站端第一重心，主动性只保留用户授权的轻回访能力。**

这条路线既符合当前 SparkCore 的技能底座与记忆层升级方向，也能自然衔接第二阶段移动端事务推进 app 的演化。

---

## 14. 一句话版本

> **Phase 1 做的是“IM 内长期陪伴 + 网站记忆控制台”，不是“泛用 AI 助手”。**
