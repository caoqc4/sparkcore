# 2026-03-27 产品层前端重构设计

## 1. 目标

本次重构面向 SparkCore 产品层中的虚拟伴侣网站前端，目标不是迁移业务逻辑，而是基于现有三层结构（技能底座、接入层、产品层）重写产品层的前端呈现与交互组织。

重构原则如下：

- 视觉与交互风格参考本地 starterkit：`/Users/caoq/git/sistine-starter-vibe-to-production`
- 控制台主结构与模块骨架参考：`doc_private/product/ai_studio_code.html`
- 保留 SparkCore 现有业务语义、路由语义与产品能力，不直接整体搬运 starterkit
- 首页与控制台一起重做，形成统一的产品层前端体验
- 设计方向采用“各半融合”：保留 starterkit 的高级感、排版节奏与组件语言，同时强化虚拟伴侣产品的关系感、陪伴感与控制台感

## 2. 总体产品模型

新版前端采用“混合式公开站 + 单角色控制台体验，底层支持多角色”的模型：

- 未登录用户访问首页，看到 landing page，首屏直接提供完整创建角色能力
- 已登录但没有角色的用户，仍停留在同一个首页首屏，并可直接创建角色
- 已登录且已有角色的用户，访问首页时直接跳转到控制台 `/app`
- 控制台默认打开最近一次交流的角色；底层支持一个用户拥有多个角色

产品结构分为三层职责：

1. 公开首页与公开页面：负责 SEO、获客、品牌表达与首次创建
2. 角色管理层：负责多角色的查看、切换、新建、编辑
3. 当前角色控制台：负责围绕一个角色进行聊天、记忆管理、集成与设置

## 3. 首页信息架构

首页 `/` 是品牌主 landing，也是主创建入口。

### 3.1 首屏 Hero

首屏采用双栏结构：

- 左侧：创建角色功能区
- 右侧：角色预览区

#### 左侧创建区

左侧承担主要转化职责，包含：

- 简洁品牌标题与说明
- 完整创建表单
- 主 CTA

创建表单字段：

- 角色类型（如亲密伴侣 / 助理型陪伴）
- 角色名称
- 语气风格（如活泼灵动 / 温柔倾听 / 沉稳理性）

CTA 根据用户状态动态变化：

- 未登录：登录并创建
- 已登录无角色：创建我的角色
- 已登录有角色：不展示该首屏，直接进入 `/app`

首屏创建区允许未登录用户先填写，再在提交时要求登录，并尽量保留已填字段。

#### 右侧预览区

右侧不做第二个表单，而做高情绪价值的角色预览区，后续为生图能力预留扩展位。

当前阶段包含：

- 默认角色主视觉 / 默认形象图
- 名字、角色类型、语气标签
- 角色一句话设定
- 1~2 条示例对话气泡
- 一个未来扩展位：生成专属形象（当前可为即将开放态）

左侧表单变更时，右侧预览同步变化，但当前阶段仅做前端映射，不引入复杂 AI 生成能力。

### 3.2 首页其余内容控制在 5 屏以内

首页剩余结构建议如下：

1. 首屏 Hero：创建角色 + 角色预览
2. 一屏角色与故事感内容区
3. Long Memory 差异点区
4. IM-native Relationship 差异点区
5. FAQ / Trust / CTA 收束区

### 3.3 首页内容策略

#### 角色与故事感区

该区最多一屏，参考同类产品如 nomi.ai 的表达方式，但不做纯照片墙。建议包含：

- 2~4 张角色视觉卡
- 每张卡配简短角色感描述
- 一段总文案，强调这不是一次性聊天，而是可以持续发展的陪伴关系

#### 差异点表达

整个首页的核心差异点统一收束为两条：

- Long Memory
- IM-native Relationship

对应内容表达应与既有 SEO 规划一致，避免 feature 过碎。

#### FAQ / Trust / CTA

首页后段保留轻量 FAQ 与信任表达，重点回答：

- 是否记得过去聊天
- 是否必须在网页聊天
- 支持哪些 IM
- 记忆能否修改或删除
- 是否私密

底部 CTA 再次回到创建角色，形成完整漏斗闭环。

## 4. 控制台总体结构

控制台需要从原型感页面升级为正式的单角色关系控制台，同时支持多角色底层能力。

### 4.1 路由与入口

建议路由结构：

- `/app`：默认角色控制台入口
- `/app/[roleId]`：指定角色控制台
- `/app/roles`：角色管理页
- `/create`：独立创建页，作为首页创建逻辑的兜底与复用入口
- `/connect-im`：独立 IM 绑定入口

其中：

- 首页 `/` 是主创建入口
- `/create` 不是主要用户心智入口，而是复用型独立创建页
- `/app` 访问时应自动解析到最近一次交流的角色
- 若没有最近交流记录，则回退到最近创建角色或默认角色

### 4.2 多角色模型

控制台默认体验仍然围绕“当前活跃角色”，但底层支持一个用户拥有多个角色。

规则如下：

- 用户可拥有多个角色
- 访问 `/app` 时默认打开最近一次交流的角色
- 用户可在控制台内切换角色
- 用户可在控制台内创建新角色
- 新角色创建成功后自动切换到该角色控制台

### 4.3 Roles 角色管理层

新增 `/app/roles` 作为角色资产管理页，职责包括：

- 展示所有角色卡片
- 查看最近活跃时间
- 标记当前默认角色 / 最近交流角色
- 新建角色
- 编辑角色基础设定
- 后续可扩展删除或归档角色

这页用于资产管理，不承担主聊天任务。

### 4.4 当前角色控制台

`/app/[roleId]` 是当前角色的主工作区，采用单角色沉浸式控制台结构：

- 左侧：角色摘要与导航
- 右侧：当前模块内容
- 顶部：轻量状态区

左侧顶部角色区既是角色摘要入口，也是快速切换入口。点击后展开角色切换面板，显示：

- 角色头像
- 角色名称
- 角色类型
- 最近互动时间
- 新建角色按钮

这样“角色切换器”负责快速切换，“/app/roles”负责完整管理。

## 5. 控制台主模块设计

控制台保留三个主模块，结构参考 `doc_private/product/ai_studio_code.html`，但重做为正式产品级 UI。

### 5.1 Web Chat

Web Chat 不只是补充对话，也可以作为完整聊天窗口使用。

定位如下：

- Web 端可进行完整聊天
- IM 仍是主生活流聊天入口
- Web 是当前角色的网页聊天与关系控制窗口

同步策略采用：**IM → Web 单向同步**。

具体规则：

- IM 端消息与回复同步到 Web 线程
- Web 端发出的消息不自动同步回 IM
- Web 页面顶部明确提示消息边界，说明：
  - IM 消息会在这里出现，用于上下文连续性
  - 网页发出的消息仅保留在 Web，不回写到 IM

这样既保留 Web 的完整聊天能力，又避免污染 IM 的主聊天流。

### 5.2 记忆中心

记忆中心是产品核心差异点之一，必须突出“可见、可修正、可删除”。

建议保留分类浏览：

- 关于我
- 关于我们
- 待跟进

每条记忆卡应包含：

- 记忆内容
- 分类
- 来源 / 更新时间
- 修改与删除操作

页面顶部要明确解释：这些记忆决定为什么角色能像“同一个人”。

### 5.3 集成与设置

该模块统一管理当前角色的：

- 模型配置
- IM 通道绑定
- 主动消息边界
- 隐私设置
- 角色边界与关系相关设定

应避免拆得过散。这里是“控制中心”的体现，应让用户在一个模块内完成角色接入生活流与边界调整。

## 6. 创建入口职责划分

创建能力采用“双入口，职责分离”的方式：

### 6.1 首页与 `/create`

负责：

- 首次创建
- 公开层 CTA 承接
- 登录回跳后的继续创建
- 快速新建角色

### 6.2 控制台内的 Roles 页

负责：

- 多角色管理
- 再创建角色
- 编辑现有角色
- 后续扩展默认角色、归档、删除等资产管理能力

### 6.3 角色控制台页

负责：

- 使用当前角色
- 聊天
- 管理记忆
- 调整集成与设置

## 7. 页面职责划分

### 7.1 Public shell

需要重做当前公开站壳层，统一：

- 顶部导航
- 页脚
- 页面容器
- section 节奏
- 背景层次
- 卡片与按钮体系

视觉语言参考 starterkit，但文案与产品表达围绕虚拟伴侣业务。

### 7.2 首页与产品主线页

优先重做：

- `/`
- `/ai-companion`
- `/ai-girlfriend`

首页使用本设计文档中的新结构，其他产品主线页复用统一 public shell 与 section 语言。

### 7.3 App shell

需要新增或重构：

- `/app`
- `/app/roles`
- `/app/[roleId]`

以及对应的角色切换器、控制台导航、状态区与主内容壳。

### 7.4 控制台模块内容层

聊天区、记忆中心、设置页均保留现有业务能力，但重写展示层、布局层与交互层。

## 8. 实现策略

实现时采用“设计语言迁移 + 现有业务逻辑复用”的方式：

- 不整体搬运 starterkit
- 复用 SparkCore 当前业务逻辑、状态判断、路由语义与产品能力
- 抽取 starterkit 中适合的设计语言、排版模式、背景层次、组件组织和动效方式
- 控制台结构以 `ai_studio_code.html` 为骨架进行产品化实现

目标是：

- 让产品层前端整体从原有页面式结构升级为统一设计系统下的公开站 + 角色控制台体系
- 保留 SparkCore 的业务内核与三层架构边界
- 为后续 SEO 主线页、角色形象生成能力、多角色管理扩展预留空间
